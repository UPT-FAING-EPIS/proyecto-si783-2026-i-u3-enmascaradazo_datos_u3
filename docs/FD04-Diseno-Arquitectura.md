## 1. INTRODUCCIГ“N

### 1.1. PropГіsito (Diagrama 4+1)

El presente documento tiene como propГіsito definir la **arquitectura software** del sistema **Enmask v2.0**, aplicando el modelo de vistas **"4+1"** de Philippe Kruchten. Este modelo permite describir la arquitectura desde mГәltiples perspectivas, cada una abordando las preocupaciones de diferentes interesados:

| Vista | Interesado Principal | PropГіsito |
|---|---|---|
| **LГіgica** | Desarrolladores | Estructura de clases, objetos y subsistemas |
| **Procesos** | Integradores de sistemas | Concurrencia, sincronizaciГіn y flujo de datos |
| **Desarrollo** | Gestores de configuraciГіn | OrganizaciГіn del cГіdigo, mГіdulos y dependencias |
| **FГӯsica** | Ingenieros de infraestructura | Despliegue en nodos, contenedores y redes |
| **Casos de Uso** | Todos los interesados | Escenarios que validan las demГЎs vistas |

El documento sirve como referencia tГ©cnica para el equipo de desarrollo, el docente del curso y los futuros mantenedores del sistema, garantizando que las decisiones arquitectГіnicas estГ©n documentadas, justificadas y trazables.

### 1.2. Alcance

El alcance de este documento abarca la arquitectura completa del sistema Enmask v2.0, incluyendo:

- **Backend:** API Gateway (FastAPI), Masking Service, Monitor Service
- **Frontend:** Dashboard HTML5 con Tailwind CSS y Chart.js
- **Persistencia:** SQLite (usuarios, mГ©tricas), 7 motores de BD destino
- **Seguridad:** AutenticaciГіn bcrypt + Google OAuth2, cifrado Fernet, gestiГіn de claves
- **Despliegue:** Docker Compose (local) y Render (nube)

## 4. ATRIBUTOS DE CALIDAD DEL SOFTWARE

Los atributos de calidad se evalГәan mediante escenarios que describen cГіmo el sistema debe comportarse bajo condiciones especГӯficas.

### Escenario de Funcionalidad

| Aspecto | DescripciГіn |
|---|---|
| **Escenario** | Un usuario autenticado ejecuta un benchmark de enmascaramiento sobre una tabla PostgreSQL con 100 filas, aplicando los 4 algoritmos (redacciГіn, hashing, encriptaciГіn, FPE). |
| **EstГӯmulo** | Solicitud POST /api/v1/execute_test con reglas de enmascaramiento |
| **Respuesta del sistema** | 1. Consulta cruda a PostgreSQL вҶ’ mide tiempo_normal_ms. 2. Aplica 4 algoritmos вҶ’ mide tiempo_mask_ms. 3. Cifra con Fernet вҶ’ mide tiempo_encrypted_ms. 4. Calcula overhead_total_ms. 5. EnvГӯa mГ©tricas al Monitor Service. 6. Retorna datos enmascarados + mГ©tricas al frontend. |
| **Medida de calidad** | Los 4 algoritmos se ejecutan correctamente; los datos enmascarados conservan la estructura original; las mГ©tricas se almacenan en SQLite. |
| **Arquitectura implicada** | main.py вҶ’ masking_service.py вҶ’ database_manager.py (PostgresDB) вҶ’ masking.py вҶ’ monitor_service.py |

### Escenario de Usabilidad

| Aspecto | DescripciГіn |
|---|---|
